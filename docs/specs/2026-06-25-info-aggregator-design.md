# 信息获取、筛选与聚合工具 — 设计文档

**Date:** 2026-06-25
**Status:** Draft

---

## 1. 项目目标

打造一个反推荐引擎式的信息获取工具，帮助使用者跳出信息茧房，获取更优质、准确、有效的多角度信息。

### 核心原则

- **多源交叉验证** — 同一话题同时搜索多个来源，对比差异
- **客观权重优先** — 内容排序基于权威度、信息密度、时效性，不做个性化推荐
- **差异可见** — 不同来源对同一内容的解读差异显式标注，而非合并成一份"统一答案"
- **使用者始终掌控** — 反馈学习的是搜索策略，不是内容偏好

### 使用场景

- 日常研究阅读（人主导搜索→阅读）
- 自动化信息管道（定时抓取订阅话题→待阅队列）
- AI Agent 后端（通过 API/MCP 挂载到 Claude 等工具）

## 2. 架构总览

```
交互层: CLI + REST API + MCP Server
   ↓
调度控制: 预算管理 / 模式选择 / 源开关
   ↓
查询改写: 单话题 → 多语言多角度查询
   ↓
源路由: 插件式分发到搜索源
   ↓ ┌──────┬──────┬───────────┬──────────┬────────┐
   Tavily  Exa    Firecrawl云  Firecrawl本地 SearXNG本地
   └──────┴──────┴───────────┴──────────┴────────┘
   ↓
标准化 + 去重: 归一化内部格式，同URL合并
   ↓
差异感知聚合: 综合摘要 + 多角度视图 + 缺口感知
   ↓
双层评分: 内容权重(客观) + 行为偏好(辅助)
   ↓
待阅队列 + 知识库
   ↓
反馈学习 → 优化搜索策略（不优化内容排序）
```

### 各层职责

| 层 | 职责 |
|---|---|
| **交互层** | 三种入口：CLI（日常用）、REST API（集成用）、MCP Server（Agent用） |
| **调度控制** | 管理各源的使用量预算、搜索模式（全开/配额优先/手动指定）、源的启用与禁用 |
| **查询改写** | 将一个人工输入的话题，根据配置自动生成中英文变体、多角度查询词 |
| **源路由** | 插件式源接口，按模式把请求分发到对应的搜索源 |
| **标准化+去重** | 不同源的返回格式统一为 StandardResult，同一 URL 合并保留最优摘要 |
| **差异感知聚合** | 三层输出：综合摘要(top-down) + 多角度视图(侧重点差异) + 原始结果列表 |
| **双层评分** | 第一层：内容自身权重（权威度/信息密度/时效性）；第二层：用户行为辅助 |
| **待阅队列+知识库** | 收件箱式阅读界面，已读归档，全文检索 |
| **反馈学习** | 基于阅读行为优化搜索策略（选源/权重分配/关键词改写），使用者可审阅调整 |

## 3. 搜索源

### 已有源（原型阶段）

| 源 | 类型 | 费用 | 核心能力 |
|---|---|---|---|
| Tavily | Cloud API | 按量 | AI 摘要 + 实时搜索 |
| Exa | Cloud API | 按量 | 语义搜索 + 论文/代码/人物 |
| Firecrawl Cloud | Cloud API | 按量 | 网页完整正文抓取 |
| Firecrawl | 本地 Docker | 免费 | 网页完整正文抓取（当前已部署） |
| SearXNG | 本地 Docker | 免费 | 元搜索（当前仅 Bing，加代理后可多引擎） |

### 插件化接口

新增搜索源只需实现：

- `SearchSource` 接口：`search(query, params) → [SearchResult]`, `health()`, `name`, `type`, `cost`
- 注册到源路由配置
- 约 50 行代码/源

## 4. 差异化聚合输出

### 输出层次

```
第1层 — 综合摘要: 各源 AI 输出(Tavily answer, Exa highlights) 的合并归纳
           + 关键共识要点
           + 各源补充的独特观点

第2层 — 多角度视图: 📰 新闻角度 (SearXNG/Bing)
                  🔬 技术深度角度 (Exa)
                  🧠 AI 总结角度 (Tavily)
                  📄 原文全文 (Firecrawl)
           + 信息缺口标注
           + 同一URL的各源解读差异

第3层 — 原始结果: 每条标注来自哪些源、全文状态、源级评分
```

### 缺口感知

聚合层主动识别信息盲区并建议补充搜索（如中英文覆盖不均、观点类型缺失）

---

## 5. 双层评分体系

### 第一层：内容自身权重（系统评定，不可被用户行为修改）

| 信号 | 说明 |
|---|---|
| 来源权威度 | 一到四级（学术/官方 → 知名媒体/行业 → 技术博客 → 社媒） |
| 作者/机构 | h-index、机构权威、领域引用量 |
| 信息质量 | 多源引用共识度、事实密度、数据引用 |
| 信息生态位 | 主流 vs 少数派、交叉验证程度、角度覆盖 |

### 第二层：用户行为（仅影响话题相关性，不影响内容权重）

- 你的标记 → 优化搜索策略（选源、权重分配、关键词）
- 不改变内容排序 ← 核心的反茧房设计

### 权限边界

```
用户行为可以改: 话题相关性、搜索策略偏好
用户行为不能改: 来源权威分级、作者评估、信息质量评分
用户手动可改: 来源权威分级规则的配置
```

---

## 6. 待阅队列

- 排序默认：信息密度 > 多源覆盖度 > 时效性（无个性化排序）
- 操作：读 / 收藏 / 丢弃 / 关联 / 深挖(+全模式搜索) / 批注
- 已读内容归档到知识库，支持全文检索和关联图谱

---

## 7. 反馈学习

- 模式识别：分析用户行为 → 优化搜索策略（非内容排序）
- 可见可审：系统产出策略调整建议，使用者确认后生效
- 学习范围：源选择、关键词改写、话题类型→源偏好映射

---

## 8. 原型阶段范围

### 第一版实现

1. **CLI 入口** — `cc-search "query" --mode budget|full|manual`
2. **源路由 + 预算控制** — 插件化源接口，可配限额
3. **标准化 + 去重** — 统一输出格式
4. **简单聚合输出** — 先做分层展示，不加缺口感知和 AI 合成摘要
5. **基本的源评分标注** — 来源权威度分级

### 后续迭代

- REST API / MCP Server
- AI 综合摘要生成（用本地 LLM 处理各源结果）
- 查询改写（多语言/多角度自动扩展）
- 待阅队列 + 知识库持久化
- 缺口感知
- 定时调度 / 订阅
- 反馈学习模块
